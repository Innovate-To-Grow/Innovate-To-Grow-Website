### 行为变化（需要知晓）

- **票务邮件链接 24 小时后失效且只能点一次。** 原来 30 天可反复点击。如果活动在发邮件 1 天后举行，用户当天点链接会看到 "Invalid or expired login link"。如不可接受，建议在签到场景改用 ticket access token（仍 30 天）而非 login token，或后续做「过期链接 → 重发新链接」的引导页。
- **退订页不再自动登录。** 点退订链接只会退订并提示成功，"Manage email preferences" 链接指向 `/account`（用户需自行登录）。
- magic login 链接默认有效期 30 天 → 7 天。

### 后续统一方案（建议，不在本 PR 内做）

目前三个流程是三种实现（DB nonce / DB token 行 / cache 标记）。建议后续收敛为一个 `apps.common`（或 `apps.authn`）里的统一一次性 token 服务：

1. **单一 `OneTimeToken` 模型**：`purpose`（ticket / magic / unsubscribe / impersonate）、`member` FK、可选 scope 对象（如 registration）、`token_hash`（salted HMAC，库里永不存明文）、`expires_at`、`used_at`、签发元数据。每个 purpose 配置自己的 TTL 与权限等级。
2. **消费语义统一**：条件 UPDATE（`WHERE used_at IS NULL AND expires_at > now`）原子认领，天然防并发重放；撤销 = 清空/标记该行。当前退订流程的 cache 标记是已知薄弱点（cache flush / 进程重启 / 本地内存 cache 多进程下都会让"一次性"失效，服务里的注释也承认了这一点）——迁到该模型即解决。
3. **最小权限分级**：action token（退订、签到核销）永不换发会话；login token 换发的 JWT 可考虑限定 scope 或缩短 access 有效期。
4. **出口防泄漏**：所有含 token 的 URL 统一经过出口清洗（本 PR 的 Gmail 导入清洗是第一例）；日志、错误上报里禁止整 URL 落盘。
5. **可观测**：`used_at` / 签发记录天然提供审计轨迹，配合现有 `LoginRateThrottle`。
